# 作品集页面文案优化建议

本文用于指导页面文案修改，目标是降低术语理解门槛，减少歧义，让 HR、面试官、技术负责人和业务方都能更快看懂项目价值。

## 总体原则

页面文案建议采用统一表达顺序：

1. 先讲业务场景：这个项目服务什么场景，解决谁的问题。
2. 再讲本人工作：我负责了什么模块、流程或工程能力。
3. 最后讲技术实现：使用了哪些框架、架构模式或工具。

目前页面整体偏“技术词先行”，读者能感受到技术密度，但需要自己推导业务价值。建议把术语翻译成业务结果，再把技术标签作为补充。

## 高优先级修改

### 1. 替换“AI辅助编程项目”

当前表达：

- `AI辅助编程项目`
- `G:\CODEX AI辅助编程项目`
- `G:\CODEX 筛选结论（AI辅助编程项目）`

问题：

容易让读者误会为“这些项目主要由 AI 生成”，削弱作者自身工程能力。同时 `G:\CODEX` 是本机路径/内部分类，不适合出现在对外页面。

建议替换为：

- `AI 协作工程实践`
- `AI 增强型项目实践`
- `AI 协作交付案例`

推荐最终文案：

```text
AI 协作工程实践
```

如果需要说明，可以补一句：

```text
这些项目展示我如何将 AI 工具纳入需求拆解、代码实现、测试验证和文档整理流程，重点仍是工程设计与交付能力。
```

### 2. 删除页面中的 `G:\CODEX`

当前表达：

```text
G:\CODEX AI辅助编程项目
G:\CODEX 筛选结论（AI辅助编程项目）
```

建议替换为：

```text
AI 协作工程实践
AI 协作项目筛选结论
```

原因：

`G:\CODEX` 对外部读者没有意义，像本地目录或临时草稿，影响专业感。

### 3. 修正“业务价”

当前表达：

```text
业务价
```

建议替换为：

```text
业务价值
```

这是明显截断问题，详情页中需要统一修正。

### 4. 重写首页首屏描述

当前表达：

```text
聚焦平台架构、业务状态机、IoT 协议服务与生产工具链，面向复杂业务系统的可交付工程能力
```

问题：

术语密度过高，“平台架构、业务状态机、IoT 协议服务、生产工具链”连续出现，对非技术读者不友好。

建议替换为：

```text
展示我在企业后端、物联网设备接入、订单流程和生产测试工具上的真实项目经验，重点体现可落地、可维护的 .NET 工程能力。
```

如果想更偏技术负责人：

```text
覆盖企业后端、IoT 设备接入、订单履约和生产测试工具，展示从业务建模、系统设计到代码落地的 .NET 工程能力。
```

## 中优先级修改

### 5. 解释能力快照中的数字

当前表达：

```text
1807 核心项目 C# 文件
55 解决方案工程规模
```

问题：

数字有冲击力，但统计口径不清楚。读者不知道 `55` 指 solution、project、模块还是仓库。

建议替换为：

```text
1807
核心项目 C# 源码文件
```

```text
55
后端、桌面端与工具工程模块
```

如果页面空间允许，可在能力快照下补充一句小字：

```text
以上数据来自本作品集选取的核心项目，用于说明项目体量与代码样本规模。
```

### 6. 替换“平台与中台架构”

当前表达：

```text
平台与中台架构
```

问题：

“中台”容易显得泛，不同公司理解不一致。

建议替换为：

```text
平台型后端架构
```

说明文案建议改为：

```text
构建可复用的后端服务层，覆盖数据访问、权限认证、业务服务和后台任务。
```

技术标签可保留：

```text
ASP.NET Core / EF Core / Repository / JWT / Identity
```

### 7. 替换“复杂业务流建模”

当前表达：

```text
复杂业务流建模
```

建议替换为：

```text
订单、仓储、审批等流程建模
```

说明文案建议改为：

```text
用状态和事件管理订单支付、仓库分配、发货检查、审批通知等多步骤流程。
```

### 8. 替换“IoT 协议与网关服务”

当前表达：

```text
IoT 协议与网关服务
```

建议替换为：

```text
设备接入与协议解析服务
```

说明文案建议改为：

```text
处理水表、仪表、产线设备等终端数据，完成协议解析、指令下发和第三方平台对接。
```

### 9. 替换“生产测试工具”

当前表达可保留，但说明可以更清楚。

建议说明文案：

```text
为产线测试和现场调试提供桌面工具，支持串口通讯、指令下发、结果校验和追溯记录。
```

## 项目命名优化

### 10. 给英文项目名补中文定位

当前项目名多数是仓库名或内部系统名，外部读者不一定知道项目类型。建议展示时采用“项目名 + 中文定位”的形式。

推荐：

```text
BoveSysApi｜智能水务 IoT 平台 API
autchon_backend｜订单履约后端
ZCYX｜企业项目管理平台
Alpaca-Link｜多协议设备接入服务
BoveMES｜生产测试工具平台
CityBound｜地信确权与边界成果系统
BoveSys.Abp｜模块化 IoT 平台实践
Ultrasonic MES｜超声表产线 MES 后端
Modbus 工业仿真套件｜现场联调与协议诊断工具
Wechat Persona Agent｜AI 用户画像系统
```

注意：

- 如果 `autchon_backend` 必须保留仓库名，可以把中文定位放在副标题。
- `Wechat` 建议统一为 `WeChat` 或中文 `微信`。
- `net5` 建议统一为 `.NET 5`。

## 详情页固定栏目建议

详情页里的栏目建议统一成以下三个：

```text
业务场景
我负责/实现的内容
技术亮点
```

不建议使用：

```text
对外展示语义
```

原因：

这个词像内部备注，不像给读者看的页面标题。

建议替换为：

```text
这个项目体现的能力
```

或：

```text
面试可讲重点
```

## AI 项目相关表达

### 11. 替换“Prompt 资产与模型调用治理”

当前表达：

```text
提示词资产与模型调用治理
Prompt 资产、模型治理与多端协同
```

问题：

“治理”偏抽象，非 AI 工程背景读者不一定理解。

建议替换为：

```text
将提示词、模型调用、错误处理和 JSON 输出格式做成可维护的后端能力。
```

项目短描述可改为：

```text
AI 用户画像系统，包含后端 API、提示词管理、模型调用封装和多端接入能力。
```

### 12. 替换“Runtime Governance”

当前架构图表达：

```text
Runtime Governance
Timeout / Error Mapping / Health
```

建议替换为：

```text
运行时稳定性
超时控制 / 错误映射 / 健康检查
```

## 工业与 IoT 项目表达

### 13. 解释“Twin 适配链路”

当前表达：

```text
Twin 适配链路
Twin 事件适配
```

问题：

“Twin”对多数读者不清楚，容易中断阅读。

建议替换为：

```text
设备数字孪生事件适配
```

或：

```text
模拟设备状态并对接数字孪生事件流
```

### 14. 解释“Modbus Master”

当前表达：

```text
Modbus Master + PLC/Robot 仿真 + Twin 适配链路
```

建议替换为：

```text
Modbus 主站采集、PLC/机器人仿真和数字孪生事件适配，服务现场联调与协议排障。
```

## 架构页文案建议

当前架构页首屏：

```text
本页聚焦可展示内容：每个项目给出业务价值、技术亮点（含标签）以及完整方法/类代码。代码块支持独立缩放
```

问题：

“可展示内容”像内部说明，“代码块支持独立缩放”是功能说明，不是价值说明。

建议替换为：

```text
从业务场景、技术亮点和关键代码三个角度展示项目能力，帮助读者快速理解每个项目解决了什么问题，以及代码如何落地。
```

按钮文案可保留：

```text
仅看关键代码
展开全部代码
```

## 修改优先级清单

优先处理：

1. 删除所有 `G:\CODEX`。
2. 将 `AI辅助编程项目` 改为 `AI 协作工程实践`。
3. 将 `业务价` 改为 `业务价值`。
4. 首页首屏副标题改成更易读的业务表达。
5. 给项目名补中文定位。
6. 将 `对外展示语义` 改为 `这个项目体现的能力`。
7. 统一 `.NET 5`、`WeChat` 等写法。
8. 对 `Twin`、`PromptOps`、`Runtime Governance` 这类术语加中文解释或替换。
