# 页面 AI 味表达清理清单

本文用于逐页清理当前作品集页面中偏 AI 腔、偏模板化、偏抽象包装的表达。修改目标不是降低技术含量，而是让文字更像真实项目说明：具体、可验证、少套话。

## 判断标准

以下表达容易显得 AI 味重：

- 连续使用抽象评价词：`体现了`、`能力`、`价值`、`亮点`、`完整`、`清晰`、`可维护`。
- 堆叠工程包装词：`闭环`、`链路`、`治理`、`协同`、`产品化`、`工程化`、`落地`。
- 用“能力总结”代替具体事实：`该项目体现了 XXX 能力`。
- 用三连形容词包装：`可落地、可扩展、可维护`。
- 技术词没有业务解释：`Prompt 资产`、`Runtime Governance`、`Twin 适配链路`。

建议写法：

1. 先说具体场景。
2. 再说做了什么。
3. 最后再列技术实现。

## 全站建议替换词

| AI 味表达 | 建议替换方向 |
| --- | --- |
| 该项目体现了……能力 | 这个项目主要处理/实现/展示…… |
| 完整交付经验 | 真实项目中的实现经验 |
| 工程化能力 | 具体写成“接口、任务、权限、数据访问、工具界面”等 |
| 产品化工程 | 产品/系统/后端服务 |
| 模型治理 | 模型调用封装、错误处理、调用记录 |
| Prompt 资产 | 提示词模板、提示词管理 |
| 协同 | 对接、配合、联动 |
| 闭环 | 关联起来、可追溯、流程完整 |
| 链路 | 流程、调用路径、数据流 |
| 可落地、可扩展、可维护 | 具体说明怎么落地、怎么扩展、怎么维护 |

## 逐页清理建议

### index.html

#### 1. 代码质量评价段

位置：`index.html`

当前表达：

```text
代码质量评价：这个项目整体采用较规范的分层结构，模块职责划分明确，依赖关系清晰。应用服务、领域逻辑与基础设施层解耦较好，具备一定的扩展性与维护性。代码中统一了异常处理、依赖注入与配置管理，常见业务逻辑具备良好的复用性。整体风格偏工程化，符合中大型 .NET 后端项目的组织方式。
```

问题：

这一整段非常像 AI 评语，几乎可套到任何 .NET 项目。缺少具体项目、具体代码、具体证据。

建议处理：

优先删除。如果保留，改成带证据的表达。

可替换为：

```text
代码样本主要来自后端 API、业务服务、数据访问、后台任务和桌面工具项目，覆盖查询分页、权限控制、状态流转、协议解析和测试结果校验等常见业务代码。
```

## 优先修改顺序

建议先改这些，因为它们最像 AI 生成总结：

1. 所有 `该项目体现了……能力/经验`。
2. `代码质量评价` 那整段。
3. WeChat Persona Agent 相关的 `Prompt 资产`、`模型治理`、`产品化工程`。
4. 所有 `闭环`、`链路`、`协同`、`治理`。
5. 所有 `可落地、可扩展、可维护` 这类三连形容词。
6. 架构页中的 `筛选结论` 评价句。
